1. Architecture Overview
The CFAPA system is a cloud-native, multi-tier intelligent pipeline deployed on the Antigravity platform with Codex providing all AI/ML inference capabilities. The architecture follows an event-driven microservices pattern with three primary planes: Data Ingestion, AI Processing, and Intelligence Delivery.

The system is designed for horizontal scalability, multi-tenancy, and model replaceability. All AI workloads are isolated from ingestion infrastructure to allow independent scaling. The Antigravity platform handles orchestration, autoscaling, secrets management, and observability. Codex provides the NLP model endpoints consumed by the AI Processing plane.

1.1 High-Level Architecture Diagram (Textual Representation)
EXTERNAL SOURCES  →  [Ingestion Layer]  →  [Event Bus]  →  [AI Processing Layer]  →  [Storage Layer]  →  [Delivery Layer]  →  END USERS / PM TOOLS

1.2 Architecture Principles
Separation of concerns: ingestion, processing, storage, and delivery are independently deployable
Event-driven: all inter-service communication via message bus (no synchronous chaining)
Model agnosticism: AI models swappable via Codex API versioning without pipeline changes
Tenant isolation: strict data partitioning per organization at storage and compute layers
Observability-first: all services emit structured logs, metrics, and traces to Antigravity's platform observability stack
Zero-trust networking: all inter-service calls authenticated via service mesh mTLS

2.1.2 Normalization Service
A stateless service that receives raw events from all connectors, applies the following transformations, and publishes normalized FeedbackItems to the message bus:
Unicode normalization (NFC form)
HTML/Markdown stripping
Language detection (fastText model)
Translation to English via Codex translation endpoint (non-English items)
PII detection and redaction (email, phone, names) via Codex NER model
Content fingerprinting (SHA-256 of normalized text) for deduplication
Metadata enrichment: source, timestamp, user segment tag, product version

2.1.3 Deduplication Store
A Redis cache (managed by Antigravity) keyed by content fingerprint, with a 30-day TTL. Normalization Service checks this store before publishing; duplicate fingerprints are discarded and logged.

2.3 Layer 3: AI Processing Layer
All AI workloads run as dedicated microservices on Antigravity, calling Codex API endpoints for model inference. Services are independently autoscaled based on queue depth.

2.3.1 Sentiment Analysis Service
Consumes feedback.normalized events. For each item, calls the Codex Sentiment Classification endpoint with a domain-fine-tuned model. Returns a 5-class label (Very Positive → Very Negative) and a confidence score. Also performs aspect-level sentiment analysis identifying polarity per product feature detected in the text.

Codex endpoint:
POST https://api.codex.ai/v1/classify/sentiment
Model: cfapa-sentiment-v1 (fine-tuned on product feedback corpus)
Latency target: p99 < 200ms

2.3.2 Theme Modeling Service
Operates on the full feedback corpus per tenant. Runs BERTopic-equivalent topic modeling via the Codex Embeddings + Clustering API. Produces theme clusters with auto-generated labels via Codex LLM summarization. Scheduled weekly or triggered on demand. Stores results in the Theme Store.

Codex endpoints:
POST https://api.codex.ai/v1/embed/batch  (generate embeddings)
POST https://api.codex.ai/v1/cluster/topics  (cluster into themes)
POST https://api.codex.ai/v1/complete  (generate theme labels via LLM)

2.3.3 Feature Request Extractor
Consumes feedback.normalized events. A Codex intent classifier identifies items containing feature requests. Matched items are embedded and grouped into feature clusters via cosine similarity. Cluster metadata is stored in the Feature Request Store.

POST https://api.codex.ai/v1/classify/intent
Model: cfapa-feature-intent-v1

2.3.4 Impact Confidence Score (ICS) Engine
Consumes feedback.sentiment.scored events and reads from Theme Store and Feature Request Store. Computes ICS for each theme and feature cluster using the weighted formula:

ICS = (Frequency_score × 0.30) + (Revenue_segment_score × 0.30) + (Sentiment_severity × 0.20) + (Recency_trend × 0.20)

Weights are tenant-configurable. ICS values are persisted to the ICS Store and trigger the top-10% alert rule evaluation.
2.3.5 Recommendation Engine
Consumes themes.refreshed events. For each top-N theme (configurable; default N=20), calls the Codex LLM completion endpoint with a structured prompt template to generate a product improvement recommendation. Recommendations include: executive summary, evidence quotes, affected segments, action type, and effort tier.

POST https://api.codex.ai/v1/complete
System prompt: cfapa-recommendation-template-v2
Max tokens: 800 per recommendation

2.3.6 Alert Engine
Polls ICS Store and Sentiment Store on a 15-minute cycle. Evaluates configurable alert rules such as: sentiment score drops >20% in 24 hours for a segment, ICS score crossing top-10% threshold, or new theme with >50 items appearing in <7 days. Publishes matched alerts to the alerts.triggered topic.

2.5 Layer 5: Delivery Layer
2.5.1 API Gateway
All client-facing requests are routed through the Antigravity API Gateway, which handles: OAuth 2.0 token validation, rate limiting (per tier), request logging, and routing to backend services. The gateway exposes:
REST API for dashboard web app and third-party integrations
WebSocket endpoint for real-time dashboard updates
Webhook receiver for inbound source push events

2.5.2 Dashboard Backend Service
Serves the web dashboard with aggregated, tenant-scoped data. Reads from Theme Store, Feature Request Store, ICS Store, and Recommendation Store. Supports date range, source, and segment filtering. Pushes real-time ICS and alert updates via WebSocket.

2.5.3 PM Integration Service
Handles bi-directional sync with Jira, Linear, and Productboard. On issue creation request from the dashboard, it calls the appropriate PM tool API with pre-formatted payload. Polls PM tools every 30 minutes to check for status changes and updates CFAPA theme status accordingly. Credentials stored in Antigravity Secrets Manager (AES-256 encrypted at rest).

2.5.4 Notification Service
Consumes alerts.triggered events. Delivers notifications via: in-app notification center (WebSocket push), email (SendGrid integration), and optional Slack webhook. Notification templates are configurable per alert type and tenant.

4.2 Weekly Theme Refresh Flow
Scheduled trigger fires (Antigravity Cron) — every Sunday 2AM UTC per tenant
Theme Modeling Service fetches all FeedbackItems from past 30 days from Normalized Feedback DB
Calls Codex Embeddings API in batch for all items not yet embedded
Calls Codex Clustering API to produce updated topic clusters
Calls Codex LLM to generate human-readable labels for each cluster
Persists updated themes to Theme Store, marking previous version as archived
Publishes themes.refreshed event to Kafka
Recommendation Engine consumes event; generates recommendations for top-20 themes
Dashboard Backend receives updated themes and recommendations; notifies PM users of new weekly digest

4.3 PM Tool Push Flow
PM user clicks 'Push to Jira' on a Recommendation Card in the dashboard
Dashboard sends POST /api/v1/recommendations/{id}/push?target=jira to API Gateway
PM Integration Service validates credentials from Antigravity Secrets Manager
Constructs Jira issue payload with recommendation summary, evidence quotes, ICS score, CFAPA ID
Calls Jira REST API POST /rest/api/3/issue
Stores Jira issue ID mapped to CFAPA recommendation ID in Integration Sync Table
Returns issue URL to dashboard; displayed as link on Recommendation Card
Every 30 minutes: PM Integration Service polls Jira for status changes; updates theme status in CFAPA if resolved

5. Security Architecture
5.1 Authentication and Authorization
CFAPA uses Antigravity's Identity Provider for all authentication. The auth flow:
Users authenticate via OAuth 2.0 Authorization Code flow with PKCE
JWTs issued by Antigravity IDP; validated at API Gateway on every request
Role-Based Access Control (RBAC) enforced at API Gateway and service layer
Three roles: Viewer (read-only dashboard), Analyst (full data access), Admin (config + integrations)

6.2 Deployment Model
All services are containerized with Docker and deployed on Antigravity's Kubernetes-backed container runtime. Deployment configurations:
Rolling deployments for stateless services (zero-downtime)
Blue-green deployments for AI model updates (instant rollback capability)
Antigravity-managed auto-scaling with HPA (Horizontal Pod Autoscaler) based on custom Kafka consumer lag metrics
Health checks: liveness and readiness probes on all services; failing services auto-restarted
Disaster Recovery: PostgreSQL automated daily backups with 30-day retention; point-in-time recovery to within 1 hour

7.2 Logging
All services emit structured JSON logs via Antigravity's logging SDK
Log levels: DEBUG (dev only), INFO, WARN, ERROR
Each log entry includes: service_name, tenant_id, trace_id, span_id, timestamp, level, message
Audit logs for all create/update/delete operations on feedback, themes, and recommendations
Audit logs are append-only; stored in S3 with 12-month minimum retention

7.3 Distributed Tracing
OpenTelemetry instrumentation on all services, exported to Antigravity's trace collector. Trace IDs propagated via W3C Trace Context headers across all inter-service calls including Codex API requests. Enables end-to-end tracing from feedback ingestion through AI processing to PM tool push.

8. External Integration Architecture
8.1 Codex AI Integration
All Codex API calls are routed through a dedicated Codex Client Library embedded in each AI Processing service. The library handles:
API key injection (retrieved from Antigravity Secrets Manager at startup)
Retry logic with exponential backoff (max 3 retries, starting at 100ms)
Request/response logging for audit purposes
Token usage tracking per tenant for billing and quota management
Model version pinning: each service specifies an explicit model version; updates require a new deployment

8.3 Inbound Source Webhooks
CFAPA exposes a webhook receiver endpoint for real-time push from supported sources:
POST https://api.cfapa.io/v1/webhooks/{connector_id}
Each request is authenticated via HMAC-SHA256 signature validation using a per-connector signing secret. Malformed or unverifiable webhook payloads are rejected with 401 and logged.

10. Key Architecture Decision Records (ADRs)
ADR
Decision
Rationale
Trade-offs
ADR-01
Use Codex for all AI inference; no direct OpenAI/Anthropic calls
Contractual alignment; Codex provides unified model management, fine-tuning, and versioning for CFAPA-specific models
Dependency on Codex availability; mitigated by Codex SLA and retry logic
ADR-02
Event-driven architecture via Kafka; no synchronous service chains
Decouples ingestion from processing; allows each AI service to scale independently; enables replay of events for model retraining
Eventual consistency; dashboard reflects state with ~15-minute lag for ICS updates
ADR-03
PostgreSQL as primary structured data store (not NoSQL)
Feedback analysis requires complex relational queries (theme ↔ item ↔ recommendation joins); strong consistency needed for ICS computation
Requires schema migrations; addressed with Antigravity managed migration tooling
ADR-04
Multi-tenant via tenant_id column (shared database); not separate schemas
Operational simplicity; Antigravity manages single cluster; sufficient isolation for v1.0
v2.0 may introduce schema-per-tenant for enterprise isolation; designed for migration path
ADR-05
ICS weights are tenant-configurable at runtime; not hardcoded
Different product teams have different prioritization philosophies; configurable weights enable product-led growth and reduces churn risk
Requires weight validation logic; wrong weights can produce nonsensical rankings — mitigated by UI guardrails
